Repository Structure

One big repo, or multiple small ones?

  • Database, back-end, front-end, map docs, geoprocessing tools, etc
  • Separation of Concerns - A repository should only have a single reason to change

Costs of splitting

  • Package files (package.json, bower.json, nuspec, setup.py)
  • Version management, code reviews
  • Syncing client and server code

Benefits of splitting

  • Code Reuse!
  • Better version tracking
  • Faster builds and deployments

Methods of Code Reuse

  1. Packaging
  2. Forking
  3. Copy/Paste

Finding Seams

  • Client / Server
  • Front-end modules (Draw tool, TOC, Filters)
  • Class libraries

Git

  • We can't check out subdirectories
  • Please don't use submodules!
